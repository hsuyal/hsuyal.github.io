By Xu Mingyang

发表于 2016-01-22

这两天研究TooleBar正好用上Style和Theme，就做了一些改变Theme的小Demo，

style文件设置两个简单的主题如下：

<!--夜间主题-->
<style name="nightTheme" parent="AppTheme">
    <item name="android:textColor">#eeeeee</item>
    <item name="android:background">#111111 </item>
    <item name="windowActionBar">false</item>
    <item name="windowNoTitle">true</item>
</style>

在每个自定义主题都要加上这两行代码，否则会报这样的错误，原因已经很清楚了。java.lang.RuntimeException: Unable to start activity ComponentInfo{com.example.xu.themedemo/com.example.xu.themedemo.MainActivity}: java.lang.IllegalStateException: This Activity already has an action bar supplied by the window decor. Do not request Window.FEATURE_SUPPORT_ACTION_BAR and set windowActionBar to false in your theme to use a Toolbar instead.

MainActivity

setTheme()的设置要在setContentView()之前，否则不会显示。 

setTheme(R.style.dayTheme);
setContentView(R.layout.activity_main);
Toolbar toolbar = (Toolbar) findViewById(R.id.toolbar);
setSupportActionBar(toolbar);

如果想做button响应改变样式，需要使用SharePreference，在点击按钮时改变SharePreference的键值对，再打开Activity时获取键值对，根据相应的键值对选择合适的style。
在setTheme之前加一个判断语句：

     preferences = getSharedPreferences("theme", Context.MODE_PRIVATE);
    if(preferences.getBoolean("theme",true)){
        //设置白天模式
        setTheme(R.style.dayTheme);
    }else{
        //设置夜间模式
        flag = false;
        setTheme(R.style.nightTheme);
    }
final SharedPreferences.Editor editor = preferences.edit();//获取编辑器

    findViewById(R.id.hello_world).setOnClickListener(new View.OnClickListener() {
        @Override
        public void onClick(View v) {
            editor.putBoolean("theme", !flag).commit();

    });

此时的更改主题需要在Acitivty重新启动时更改，但是如果不重新启动Activity的时候，就是对该需要换主题的Activity的所有背景和字体等操作换一下属性就ok了，不需要对每一个Activity进行操作。但是条件复杂的情况下需要遍历每一个view，暂时不知道更简单的不通过重启Activity的方法去更换主题，因为主题在setCOntentView()之后没有效果，所以暂时不知道方法，作为笔记吧先放到这里，以后补充。

By Xu Mingyang

发表于 2016-01-21

今天用了一个新的东西做了一个toolbar，我之前的toolbar都是自己写的view，tab页面也都是自己写的view，后来tabs页面换成了viewpager，但是toolbar一直还是用的view，今天我给改成了android.support.v7.widget.Toolbar做toolbar

Toolbar是一个专门做tool的东东，其中封装了很多好东西，比我之前自己写确实要方便很多。再加上PagerSlidingTabStrip，Fragtment和Viewpager基本上一个app框架就搭好了，特别是在包含好几个tab页面的app中，我现在记录下我写的过程，以免后来忘记供自己随时查阅。

android.support.v7中几个用到的类

  1. ActionBarDrawerToggle官方的解释是：This class provides a handy way to tie together the functionality of
    {@link android.support.v4.widget.DrawerLayout} and the framework ActionBar to implement the recommended design for navigation drawers.
  2. ShareActionProvider分享控件
  3. SearchView搜索控件，功能很强大

<menu xmlns:android="http://schemas.android.com/apk/res/android"

    >
</item>
<item android:id="@+id/action_settings"
    android:title="@string/action_settings"
    android:orderInCategory="100" app:showAsAction="never" />

</menu>

关于包含app:actionProviderClass=”android.support.v7.widget.ShareActionProvider和app:actionViewClass=”android.support.v7.widget.SearchView这种系统自带的东东的在onOptionsItemSelected方法中是无法获取id的，但是如果自己写就可以获取到id，如给他设置一个一样的android:icon就可以自己定义分享功能。

我们需要在onCreateOptionsMenu这个方法中做处理：代码如下

@Override
public boolean onCreateOptionsMenu(Menu menu) {
    // Inflate the menu; this adds items to the action bar if it is present.
    //对ShareActionProvider的处理,MenuItemCompat是MenuItem的兼容版
    getMenuInflater().inflate(R.menu.menu_main, menu);
    mShareActionProvider = (ShareActionProvider) MenuItemCompat.getActionProvider(menu.findItem(R.id.action_share));
    Intent intent = new Intent(Intent.ACTION_SEND);
    intent.setType("text/*");
    mShareActionProvider.setShareIntent(getShareIntent());

    //以下是对SearchView的处理
    MenuItem search_item = menu.findItem(R.id.ab_search);
    SearchView mSearchView = (SearchView) MenuItemCompat.getActionView(search_item);

    return super.onCreateOptionsMenu(menu);
}

private Intent getShareIntent(){
    Intent intent = new Intent(Intent.ACTION_SEND);
    intent.putExtra(Intent.EXTRA_TEXT, "Share Words");
    intent.setType("text/plain");
    return intent;
}

需要注意的是：mShareActionProvider.setShareIntent(getShareIntent());可能找不到setShareIntent这个方法，这个问题困扰我蛮久，但是很简单，因为ShareActionProvider不止在v7中有，v4中也有，貌似非这两个类中也有一个，所以import的时候要特别注意，不要import错了。否则怎么都找不到撒。好了今天该回去了，貌似就是贴代码了，木办法，还是菜鸟一个，慢慢学吧。

WebView打开本地图片

By Xu Mingyang

发表于 2016-01-20

使用WebView打开本地图片的一些小问题

WebView Android中默认不支持上传文件，不同版本的WebView有区别，因此需要对不同版本适配，这里只简单介绍android5中的使用，至于6和更低版本的，只要在WebChromeClient中重写openFileChooser就可以；

首先在Activity中声明

private WebView webView;
private ProgressDialog progressBar;
private ValueCallback<Uri[]> mUploadMessage;
private final static int FILECHOOSER_RESULTCODE = 1;

接着在Activity中找到WebView并设置

webView.getSettings().setJavaScriptEnabled(true);//设置可以和JS交互
webView.setWebChromeClient(new MyWebChromeClient());//设置打开文件和load进度发生变化时的回调
webView.setWebViewClient(new MyWebClient());//设置不跳转至系统默认浏览器，就从该webView中打开
webView.loadUrl("http://www.script-tutorials.com/demos/199/index.html");//此处使用的链接是在网上找到的别人写好的链接，可能随时会不能用

创建一个WebViewClient

 class MyWebClient extends WebViewClient {
    @Override
    public void onPageStarted(WebView view, String url, Bitmap favicon) {
        super.onPageStarted(view, url, favicon);
        Log.i("UP_LOAD","111111111111111");
    }

    @Override
    public boolean shouldOverrideUrlLoading(WebView view, String url) {
        view.loadUrl(url);
        return true;
    }

    @Override
    public void onPageFinished(WebView view, String url) {
        super.onPageFinished(view, url);
        Log.i("UP_LOAD","4444444444444----------->");
       progressBar.hide();
    }
}

WebViewClient的一些方法说明，webView主要是帮助处理各种通知和请求事件的，和WebChromeClient的区别在于如果只是用来处理一些html页面的内同，只用WebViewClient就足够了，但是如果需要丰富的效果，如上传文件、JS等，就要用到WebChromeClient。

  1. 点击请求链接时使用，重写该方法返回true代表跳转至系统默认浏览器还是从该webView中跳转；
    `public boolean shouldOverrideUrlLoading(WebView view, String url) { 
    view.loadUrl(url);
return true;}`
  2. 在页面加载时开始调用,和WebChromeClient中的onProgressChanged相比回调晚
    public void onPageStarted(WebView view, String url, Bitmap favicon) {super.onPageFinished(view, url);}
  3. 在页面加载结束时调用，和WebChromeCLient中onProgressChanged相比回调晚
    `public void onPageFinished(WebView view, String url) {
        super.onPageFinished(view, url);
}`

创建一个WebChromeClient

public class MyWebChromeClient extends WebChromeClient {

    //此函数的onProgressChanged函数触发早于WebViewClient但WebViewClient的结束时间晚于该函数，
    // 所以在该函数中开始，在WebViewClient函数中结束.不论在setVebViewClient和SetWebChromeClient的前后
    @Override
    public void onProgressChanged(WebView view, int newProgress) {
        if (newProgress < 100){
            Log.i("UP_LODE","2222222---------------->"+newProgress);
            progressBar.setMessage("加载中");
            progressBar.show();
        }else{
            Log.i("UP_LOAD","333333----------->");
        }
    }

    //Android 5.0+
    @Override
    public boolean onShowFileChooser(WebView webView, ValueCallback<Uri[]> filePathCallback, FileChooserParams fileChooserParams) {
        if (mUploadMessage != null) {
            mUploadMessage.onReceiveValue(null);
        }

        mUploadMessage = filePathCallback;
        Intent i = new Intent(Intent.ACTION_GET_CONTENT);
        i.addCategory(Intent.CATEGORY_OPENABLE);
        i.setType("*/*");

        startActivityForResult(Intent.createChooser(i, "File Chooser"), FILECHOOSER_RESULTCODE);
        return true;
    }
}

以上只能针对android5，若要适配android3.0+需要重写以下代码

public void openFileChooser(ValueCallback<Uri> uploadMsg) {

if (mUploadMessage != null) {
    mUploadMessage.onReceiveValue(null);
}
mUploadMessage = uploadMsg;
Intent i = new Intent(Intent.ACTION_GET_CONTENT);
i.addCategory(Intent.CATEGORY_OPENABLE);
i.setType("*/*");
startActivityForResult(Intent.createChooser(i, "File Chooser"), FILECHOOSER_RESULTCODE);
}

若要适配Android 4.1需重写以下代码

public void openFileChooser(ValueCallback<Uri> uploadMsg, String acceptType, String capture) {

if (mUploadMessage != null) {
    mUploadMessage.onReceiveValue(null);
}
mUploadMessage = uploadMsg;
Intent i = new Intent(Intent.ACTION_GET_CONTENT);
i.addCategory(Intent.CATEGORY_OPENABLE);
i.setType("*/*");
startActivityForResult(Intent.createChooser(i, "File Chooser"), FILECHOOSER_RESULTCODE);
}

我的机器上只装了Android5所以低版本的并未验证，但是应该是可靠地，毕竟这个比较简单，在Android5中openFileChooser已经被丢弃，so呢，重写就好了。

如果想触发进度条，我的理解是在WebChromeCLient中的onProgresChanged中触发显示进度条，在WebVIewClient中的onPageFinished中结束进度条。

不知道这样理解有没有问题，至少在我的程序中这样没出错而且是比较靠谱的。

还有一个问题需要注意：由于不同版本的差别，Android 5.0以下的版本，ValueCallback 的onReceiveValue接收的参数类型是Uri, 5.0及以上版本接收的是Uri数组，在传值的时候需要注意。

接着需要在Activity中回调onActivityResult

处理接收到的文件，传给网页响应，前面我们使用startActivityForResult打开文件，该方法中接受结果

   @Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {
    super.onActivityResult(requestCode, resultCode, data);
    if (requestCode == FILECHOOSER_RESULTCODE) {
        if (null == mUploadMessage)
            return;
        Uri result = (data == null || resultCode != RESULT_OK) ? null: data.getData();
        if (result != null) {
            mUploadMessage.onReceiveValue(new Uri[]{result});
        } else {
            mUploadMessage.onReceiveValue(new Uri[]{});
        }
        mUploadMessage = null;
    }
}

即使获取的结果为null，也要传给web，即直接调用mUploadMessage.onReceiveValue(null),否则网页会阻塞。
注意混淆文件的时候onFileChooser已经被抛弃需要特别设置该方法不混淆，否则将无法选择文件。
